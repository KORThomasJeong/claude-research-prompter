# 유형 1 — 기술 검증형 (Tech Due Diligence)

**용도**: 기술/도구/아키텍처 선정, ADR 근거 수집, PoC 전 사전조사
**전형적 상황**: "벡터DB 뭘 쓸까", "이 아키텍처 결정의 근거가 필요해", "A vs B vs C 비교"

## 컨텍스트 수집 (부족한 것만 질문)

1. 비교 대상 후보는? (없으면 후보 발굴부터 리서치 범위에 포함)
2. 결정 기준 중 양보 불가 조건은? (예: 온프레미스 필수, 특정 클라우드, 예산 상한)
3. 적용 환경 규모는? (사용자 수, 데이터량, 트래픽)
4. 보안/컴플라이언스 제약은? (사내망, 데이터 주권, 인증 요건)
5. 이 결정의 되돌림 비용은? (락인 민감도)

## 기본 MECE 프레임

- **차원**: 후보 기술/도구/아키텍처 옵션 (3~5개)
- **관점**: ① 아키텍처 적합성·통합성 ② 성능·확장성(벤치마크) ③ 보안·컴플라이언스
  ④ 운영 부담·TCO ⑤ 생태계 성숙도(커뮤니티, 릴리즈 주기, 이슈 대응)
  ⑥ 락인 리스크·마이그레이션 경로

## 브리프 작성 시 요구할 분석 방식

- 각 후보의 설계 철학 차이가 어떤 운영상 결과로 이어지는지 인과적으로 설명
- 공식 벤치마크와 제3자 벤치마크를 구분하고 측정 조건 명시
- "우리 제약 조건" 하에서의 탈락 사유를 먼저 판정한 후 생존 후보를 심층 비교
- GitHub 이슈/포럼에서 드러나는 실제 운영 문제(failure mode) 수집
- 2년 후 시나리오: 각 후보 선택 시 예상되는 기술 부채

## 출력 규격

1. 결정 기준별 가중치 제안 + 후보 비교표 (근거 출처 링크 포함)
2. 각 후보의 치명적 약점 1개씩 (deal-breaker 후보)
3. 권고안 + 반대 시나리오 ("이런 조건이면 2순위가 낫다")
4. ADR 초안 (Context / Decision / Consequences / Alternatives Considered)

## 검증 루프 (2차 프롬프트)

- "권고안 [X]를 채택했다가 실패한 사례를 찾고 실패 원인을 분석하라"
- "비교표의 각 수치에 대해 출처의 측정 조건과 우리 환경의 차이를 평가하라"
- "이 비교에서 누락된 평가 기준이 있다면 무엇이고 왜 중요한가"
